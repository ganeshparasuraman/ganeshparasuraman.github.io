The Setup

Part 1 left a question unanswered.

In Part 1 we established that a Digital Twin of an Organisation (DTO) needs two things working together: a structural layer (capabilities, systems, teams) and an operational layer (live events, state changes, failures). The operational layer is what makes the DTO useful in real time.

But we glossed over something critical: where does that operational data actually live, and how does it get from the transactional system that writes it to the analytical system that reads it? For forty years, the answer was: a pipeline. ETL, CDC, replication. A copy of the data, with a lag, and a team responsible for maintaining the bridge.

Every DTO we have built has a hidden seam: the gap between the database that captures business events and the engine that reasons over them. That seam has a name — it is called the data pipeline — and it is the most fragile part of the architecture.

One major shift happened in June 2026 that forced a rethink. Databricks, at its Data + AI Summit, announced LTAP — an architecture that claims to have finally closed it. It is directly relevant to anyone building a DTO.

Enter LTAP

Databricks' answer: unify at the storage layer, not the engine layer.

On June 16 2026, Databricks announced LTAP — Lake Transactional/Analytical Processing. The core insight is subtle but important: previous unification attempts tried to make one query engine handle both workloads. LTAP keeps dedicated engines (Postgres for transactions, Spark/Lakehouse for analytics) but makes them share a single copy of the data in open formats on object storage.

Key claim
✓ Transactional writes land in Delta/Iceberg at the point of write. The Lakehouse reads the same copy. There is no ETL step, no CDC connector, no schema mapping job to maintain — and no replication lag between the system of record and the system of reasoning.

Databricks also announced Lakehouse//RT alongside LTAP — a real-time analytics engine that delivers sub-100ms query latency directly on Delta and Iceberg tables, handling up to 12,000 queries per second. This removes the last reason to maintain a separate real-time serving tier next to the lakehouse.

The honest caveat: LTAP was announced at Data + AI Summit 2026 and is listed as "coming soon." No independent benchmarks of the single-storage claim under high-write OLTP load exist yet. The production proof is still emerging.

The Landscape

HTAP, zero-ETL, LTAP — what is actually different?

Three architectures now compete for the "unified" label. They differ fundamentally in where they attempt unification and what tradeoffs that creates for DTO builders.

Three unification approaches — where each closes the seam
HTAP (Engine-level)
One engine handles both workloads. Row + columnar storage maintained in parallel.
Examples: SAP HANA, TiDB, SingleStore
Write amplification: every commit writes to both stores
Resource contention: OLAP scans spike OLTP latency
Isolation: none without physical separation (which reintroduces lag)
⚠ Two engines hidden inside one product. Neither performs at full capability.
Zero-ETL (Replication-level)
Separate engines, automated CDC pipeline between them. Marketed as "zero ETL" — still two copies.
Examples: Aurora → Redshift, ClickHouse + ClickPipes CDC
Replication lag: seconds to minutes
Governance: two catalogs to keep aligned
Operational cost: CDC connectors break on schema change
Best current practice — the seam still exists, but manageable with good schema discipline and pipeline observability.
LTAP (Storage-level)
Separate engines, single copy of data in open formats. No pipeline between them.
Example: Databricks LTAP (Lakebase + Lakehouse)
Lag: zero — transactional write IS the analytical record
Governance: Unity Catalog, single identity model
Tradeoff: currently Databricks-only; production benchmarks emerging
✓ Architecturally closes the seam. Awaiting production validation at scale.

ClickHouse's unified data stack (Postgres + ClickHouse + ClickPipes CDC) is the strongest zero-ETL implementation available today — sub-second replication, tight schema contracts, query federation back to Postgres. For most DTO teams not on Databricks, this is the right answer right now.

What Changes for the DTO

How this reshapes the five DTO building blocks.

In Part 1 we described five components every DTO needs: event streaming, graph-aware models, unified metadata, real-time query layers, and AI agents. The storage unification shift touches each one — some profoundly, some not at all.

AI
AI Agents — the primary motivation for LTAP
Agents do not tolerate pipeline lag. A reasoning system acting on stale data acts on a lie. Databricks' framing: "AI agents, not people, are now the database's primary users."
Human-speed analytics
Minutes-old data is fine for a dashboard. The seam is tolerable.
→
Agent-speed reasoning
Stale data = wrong action
An agent routing a customer, triggering an order, or flagging a risk must read the current commit — not the last replication cycle.

The Verdict

Four things that actually changed. One that did not.

What storage-level unification changes for DTO builders
Changed ✓
The pipeline is no longer inevitable
For the first time there is a credible architecture where a transactional write IS the analytical record. The structural seam between OLTP and OLAP is no longer a law of physics — it is a choice.
Changed ✓
Governance becomes a first-class citizen
A single storage copy governed by one catalog means lineage, permissions, and quality rules apply once. The dual-catalog drift problem — where OLTP and OLAP metadata diverge — is eliminated by design.
Changed ✓
Agents can reason on the current commit
An AI agent grounded on unified storage reads business state as of the last write, not the last replication cycle. For DTOs driving autonomous decisions, this is the difference between a nervous system and a history book.
Changed ✓
Real-time serving tier is optional
Lakehouse//RT at sub-100ms on governed Delta/Iceberg tables removes the need for a Redis/Druid serving layer in most DTO query patterns. Fewer moving parts. Better cost profile.
Did not change ⚠
The hard part of a DTO is still the hard part
Graph modelling. Business event schema design. Connecting operational data to the structural architecture maintained by Enterprise Architects. Defining wellness signals. Building the three-question capability stack — observability, impact traversal, intelligence. None of this is solved by storage-layer unification. LTAP gives the DTO a better foundation. It does not build the DTO for you. The engineering discipline described in Part 1 remains exactly as important.

The storage war is real, and for DTO builders it matters. But the most important architectural decisions — how you model business state, how you trace propagation, how you expose consequence queries to AI agents — are independent of whether the data pipeline between your transactional and analytical systems has zero milliseconds of lag or two seconds. Build the DTO correctly. Then let the storage layer catch up.

LTAP does not make the DTO easier to conceive. It makes the DTO possible to trust — because the operational truth it reflects is finally current.
