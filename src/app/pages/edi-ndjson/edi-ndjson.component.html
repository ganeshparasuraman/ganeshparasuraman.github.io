The problem with EDI

If you've worked anywhere near US healthcare billing, you've met X12 EDI — the decades-old electronic data interchange format that carries claims (837) and remittances (835) between providers, clearinghouses, and payers. A single file looks like this: 

ISA*00*          *00*          *ZZ*PAYER123456789 *ZZ*PROVIDER98765  *230101*1200*^*00501*000000905*0*P*:~
GS*HP*PAYER123*PROVIDER987*20230101*1200*1*X*005010X221A1~
ST*835*0001~
BPR*C*1500.00*C*ACH*CCP*01*111000025*DA*123456789~
CLP*PAT001*1*1500.00*1350.00*150.00*HM*PAYER-ICN-001*11*1~
CAS*CO*45*100.00**253*50.00~
SVC*HC:99213*250.00*200.00**1~

It's compact, positional, and deeply nested — segments group into loops, loops nest under hierarchical levels, and the meaning of an element depends entirely on which segment and which loop it lives in. It is also almost impossible to query directly. Analysts want columns and rows; EDI gives you tilde-delimited segments with colon-separated composites and a 300-page implementation guide.

The goal: take any 835 or 837 file and flatten 100% of its data into a format a data engineer can actually query — without writing a bespoke parser for every segment type.

What we set out to build

A Spring Boot service that accepts an EDI file and emits NDJSON (newline-delimited JSON) — one JSON object per EDI segment, each tagged with its segment type, loop context, and the correlation keys that stitch the document hierarchy back together. One line per segment means downstream tools can filter with grep, load with pl.read_ndjson(), or query with DuckDB's read_ndjson_auto().

A sample output line for a claim-payment segment:

{"segment_type":"CLP","loop_context":"2100","isa_control_number":"000000905",
 "st_control_number":"0001","clp01_patient_control_number":"PAT001",
 "clp03_total_claim_charge_amount":1500.0,"clp04_claim_payment_amount":1350.0,
 "source_filename":"sample_835.edi","parsed_at":"2026-06-20T08:11:11Z"}

Parsing X12 with Smooks

Rather than hand-roll a tokenizer, we use a custom SAX XMLReader that streams each EDI segment as an event, feeding it through the Smooks pipeline. The reader detects delimiters straight from the ISA header (element, component, repetition, and segment terminators are declared in fixed positions of the interchange envelope), so it adapts to whatever separators a trading partner uses.

Each segment surfaces as a SegmentEvent exposing positional accessors — getElement(2), getComponent(1, 0) for composite sub-elements. The parser core never changes; everything that varies between segment types lives in configuration.

Config-driven segments

Every segment type is described by a YAML file. The Avro/JSON field layout and type coercion are derived from this config at startup — no Java code per segment. Adding a new segment is a drop-in file plus a restart. 

segment: CLP
description: Claim Payment Information
transactionSets: [835]
correlationKeys:
  - isa_control_number
  - st_control_number
fields:
  - { name: clp01_patient_control_number, position: 1, type: string,
      setsCorrelationKey: clp_patient_control_number }
  - { name: clp03_total_claim_charge_amount, position: 3, type: decimal }
  - { name: clp04_claim_payment_amount,      position: 4, type: decimal }

Field types map cleanly to JSON: string, integer, decimal, plus composite (flattened into multiple columns) and repeatingGroup (more on that below). The service ships with 25 segment configs covering the common envelope plus 835 and 837 segments.

Correlation keys: stitching the hierarchy

An EDI document is a tree: ISA → GS → ST → CLP → SVC for remittances, ISA → GS → ST → HL → CLM → LX for claims. Every flattened row needs the breadcrumb back to the root so the pieces can be re-joined downstream. A CorrelationKeyTracker walks the segment stream and maintains the current "where am I in the tree" pointer — and crucially, clears stale child keys when a parent changes, so one claim's service-line numbers never bleed into the next.

We pushed even this logic into config. The cascade tree is declared as explicit leaf-to-root paths:

paths:
  # 837: ISA → GS → ST → HL → CLM → LX
  - [isa_control_number, gs_group_control_number, st_control_number,
     hl_id, clm_patient_account_number, lx_line_number]

  # 835: ISA → GS → ST → CLP → SVC
  - [isa_control_number, gs_group_control_number, st_control_number,
     clp_patient_control_number, svc_line_number]

Setting any key clears everything to its right in any path containing it. The runtime is a six-line loop, and supporting a brand-new transaction set (270/271, 834, …) is a single new line — no Java change.

Repeating groups: the CAS segment

Some segments pack repeating tuples into their positions. The Claim Adjustment Segment (CAS) — which explains every adjustment dollar — carries up to six (reason code, amount, quantity) triples across positions 2–19. Writing 19 fields by hand is miserable, so we added a repeatingGroup directive: 

  - name: cas_adjustment
    type: repeatingGroup
    startPosition: 2
    repeat: 6
    stride: 3
    group:
      - { name: reason_code, offset: 0, type: string }
      - { name: amount,      offset: 1, type: decimal }
      - { name: quantity,    offset: 2, type: decimal }

This expands into flat columns — cas_adjustment_1_reason_code, cas_adjustment_1_amount, … through occurrence 6 — with unused slots emitted as JSON null. The same directive will later cover PLB (stride 2) and the 837 diagnosis segment HI.

Why this matters A CAS row carries the parent claim's clp_patient_control_number and the service line's svc_line_number, so every adjusted dollar joins straight back to its claim and line in a downstream SQL query — no manual EDI walking required.

Pluggable cloud I/O

The service started life writing to local disk. To make it production-ready, both input and output became configurable cloud locations behind a single StorageClient interface with four implementations:

BackendSDKCredentials
S3AWS SDK v2access key/secret + optional session token; endpoint override for MinIO
GCSgoogle-cloud-storageservice-account JSON or application-default
AZUREazure-storage-blobaccount key, SAS token, or connection string
LOCALjava.nionone — also the hermetic test path

A job is described by a JSON document in MongoDB. The service reads the EDI from the input location, parses it to a local staging file, then uploads the NDJSON and a meta.json manifest to the output location — so you can read from S3 and write to GCS in the same job: 

{
  "id": "claims-daily-s3-to-gcs",
  "ediType": "auto",
  "input": {
    "type": "S3", "bucket": "acme-edi-inbound",
    "path": "835/2026-06-20/remit.edi",
    "credentials": { "region": "us-east-1",
                      "accessKeyId": "env:AWS_ACCESS_KEY_ID",
                      "secretAccessKey": "env:AWS_SECRET_ACCESS_KEY" }
  },
  "output": {
    "type": "GCS", "bucket": "acme-edi-curated", "path": "ndjson/835",
    "credentials": { "serviceAccountJson": "env:GCP_SA_JSON" }
  }
}
Design note The parser core stays oblivious to the cloud. Input arrives as an InputStream; output is staged to a local temp file and uploaded on completion. That single seam kept the Smooks pipeline, trackers, and NDJSON writer entirely unchanged when we added cloud support.

Secrets & credentials

Credential values are never required inline. Any value can be a reference — env:NAME pulls from an environment variable, config:dotted.key from Spring config — resolved at runtime by a pluggable SecretResolver. A future HashiCorp Vault resolver (prefix vault:) drops in with zero call-site changes. When a config is read back through the API, inline literals are masked to *** while harmless references are shown as-is.

Using the API

The whole lifecycle is four endpoints:

Method & pathPurpose
POST /api/v1/edi/configsRegister a job config (credentials masked on read)
POST /api/v1/edi/parse?configId=…Run a job — cloud-in → parse → cloud-out
GET /api/v1/edi/jobsList completed runs
GET /api/v1/edi/jobs/{id}/segments.ndjsonStream the NDJSON back from its output location

A run returns segment counts and the output URI:

# Trigger a parse
curl -X POST "http://localhost:8082/api/v1/edi/parse?configId=claims-daily-s3-to-gcs"

# → { "ediType": "835", "totalRecords": 19, "outputType": "GCS",
#     "outputUri": "gs://acme-edi-curated/ndjson/835/<jobId>/segments.ndjson", ... }

And querying the result is finally pleasant:

import polars as pl
df = pl.read_ndjson("segments.ndjson")
df.filter(pl.col("segment_type") == "CAS") \
  .group_by("cas01_group_code") \
  .agg(pl.col("cas_adjustment_1_amount").sum())
# CO → 150.00,  PR → 50.00   — every adjustment dollar, attributed

Takeaways

  • Push variation into config. Segment layouts, the correlation-key cascade, and storage backends are all data, not code. New segments and transaction sets are YAML edits + a restart.
  • Keep one stable parser core. A single InputStream-in / staged-file-out seam let us bolt on MongoDB and three cloud SDKs without touching the Smooks pipeline.
  • Make the hard format joinable. Correlation keys on every row turn a nested, positional document into flat tables that re-join with a plain SQL JOIN.
  • Test the abstraction, not the cloud. The LOCAL backend exercises the exact cloud-in/cloud-out wiring with no credentials, keeping the suite hermetic.

The result: drop an 835 or 837 anywhere — a bucket, a blob, a local path — point a JSON config at it, and get back clean, queryable NDJSON with every segment, every adjustment, and every correlation key intact. Built with Java 21 · Spring Boot 3.2 · Smooks · MongoDB · AWS / GCP / Azure SDKs.
